R&R Importaciones es un sistema SaaS multi-tenant para agencias de importación de vehículos en la frontera norte de México. Gestiona el ciclo de vida completo de importación: desde la cotización al cliente hasta el cobro final, pasando por trámites aduanales, pedimentos, pagos, gastos, y personal de campo.

Cliente fundador: R&R Importaciones (Nuevo Laredo, Tamaulipas). Arquitectura multi-tenant pero operación inicial single-tenant.

✅ Roadmap MVP completo al 100%. Las 7 fases del plan original (Fase 0 + Camino A (3) + Camino B (2) + Integración) están implementadas. Además se agregaron 4 funcionalidades extra no previstas: Portal del Cliente, Módulo Campo, Rodri (asistente IA), y Plantillas de Mensaje.

🏗️ Arquitectura General

🎨 Frontend

  • Framework: Angular 19.2
  • Componentes: Standalone (sin NgModules)
  • Estado: Signals + RxJS
  • UI: PrimeNG 19 + Tailwind CSS v4
  • Design System: Tokens CSS personalizados (design-system.css)
  • Gráficas: ECharts 6
  • Tiempo Real: SignalR (@microsoft/signalr 10)
  • Drag & Drop: Angular CDK
  • Fuentes: Inter Variable (body), JetBrains Mono (datos)

⚙️ Backend

  • Runtime: .NET 10
  • Arquitectura: Clean Architecture (4 capas)
  • ORM: EF Core 10 + Npgsql (híbrido con Dapper)
  • Auth: JWT Bearer + Refresh Tokens + BCrypt
  • Validación: FluentValidation (parcial)
  • Logs: Serilog (archivo + consola)
  • Tiempo Real: SignalR Hubs
  • PDF: QuestPDF 2026.5
  • Email: MailKit 4.16
  • Storage: Local (desarrollo) / Cloudflare R2 (producción)

🗄️ Base de Datos

  • Motor: PostgreSQL 16 (Neon Cloud)
  • Extensiones: pgcrypto, pg_trgm
  • Conexión: Neon Serverless (AWS us-east-1)
  • Migrations: EF Core (proyecto separado RR.Migrations)
  • SSL: Requerido
  • Naming: Tablas en español, columnas snake_case

📦 Módulos por Área de Negocio

🔐 Autenticación y Seguridad

  • Login con username/password + BCrypt
  • JWT con HMAC-SHA256, expiración 24h (configurable)
  • Refresh tokens con rotation (7 días de vida)
  • 5 roles: ADMIN, GERENTE, OPERADOR, CAPTURISTA, CAMPO
  • 22 permisos granulares: TRAMITES_VER, TRAMITES_EDITAR, CLIENTES_VER, COTIZACIONES_CREAR, PAGOS_VER, REPORTES_FINANCIEROS, USUARIOS_VER, etc.
  • Auth vía query string para SignalR connections

👤 Clientes

  • Identificación por APODO (ej. "DOMINGO", "LUIS LDO TX", "KIKO")
  • Apodo único por tenant (case-insensitive)
  • No se puede cambiar apodo si ya tiene trámites
  • Soft delete: no permitido si tiene trámites activos
  • Procedencia, tipo persona (FÍSICA/MORAL), RFC

🚗 Vehículos

  • VIN de 17 caracteres, único por tenant
  • VINCorto = últimos 6 caracteres para búsqueda visual
  • Auto-clasificación de fracción arancelaria por cilindrada + categoría
  • Checkpoints pre-trámite: cumplió requisitos, sello aduanal, pedimento proforma
  • No se puede crear vehículo sin cliente asociado
  • Si VIN ya existe, sugiere el existente
  • Soft delete: no permitido si tiene trámites

📋 Trámites

  • Número consecutivo secuencial por tenant (formato RR-XXXX)
  • Tipos: NORMAL, EXPRESS (con cargo extra), ASESORIA_LOGISTICA
  • Máquina de 16 estados con validación de transiciones
  • Cobro_total no modificable si ya tiene pagos
  • No puede pasar a VERDE_ENTREGADO sin entrega registrada
  • No puede pasar a COBRADO sin pagos ≥ cobro total
  • AMARILLO se asigna automáticamente: VERDE + saldo>0 + 7 días
  • Pedimentos: 1 original + N rectificaciones (R1/R2/R3)
  • Documentos requeridos por tipo con estatus PENDIENTE/RECIBIDO/VALIDADO

💰 Cotizaciones

  • Folio formato COT-YYYYMM-XXXX
  • Estados: BORRADOR → ENVIADA → ACEPTADA/RECHAZADA → CONVERTIDA
  • Solo BORRADOR puede editarse
  • Solo ACEPTADA puede convertirse a trámite
  • Expiran a los 7 días (background job automático)
  • VIN decodificado via NHTSA (cache 30 días)
  • Precio: lookup jerárquico Específico → Marca → Genérico
  • Antigüedad: min 1 año, max 12 años (clamped)
  • TC: Banxico FIX + margen configurable (default 0.30 MXN/USD)
  • Match scoring con ajuste por cilindros
  • PDF profesional generado con QuestPDF, envío email + WhatsApp deep link

💳 Pagos

  • Comprobante OBLIGATORIO (jpg/png/pdf, max 10MB)
  • Pagos en USD requieren tipo de cambio
  • Verificación manual por GERENTE/ADMIN
  • Bulk verification disponible
  • Si pagos verificados ≥ cobro_total → trámite pasa a COBRADO
  • Recibo PDF auto-generado (formato REC-YYYYMM-XXXX)
  • Soft delete: solo si NO verificado
  • Métodos: TRANSFERENCIA, EFECTIVO, DEPOSITO, CHEQUE

🐜 Gastos Hormiga

  • Gastos operativos durante trámites (gasolina, fletes, permisos, etc.)
  • Categorías: COMBUSTIBLE, TRANSPORTE, DOCUMENTACION, SERVICIOS, OTROS
  • Se_carga_al_cliente → aparece en saldo pendiente del trámite
  • Pueden vincularse a cliente, vehículo, y/o trámite
  • Soft delete aplica

🌾 Campo (Yardero)

  • Tipos de tarea: FOTOS_YARDA (default)
  • Estatus: ABIERTA → (tomada) → COMPLETADA
  • Fotos: max 15MB, jpg/jpeg/png/webp
  • Confirmación de VIN en campo
  • Notificaciones SignalR en tiempo real
  • Personal de campo con roles: ENTREGADOR, CHOFER, AMBOS

🤖 Rodri (Asistente IA)

  • Chat interno con Gemini API, solo ADMIN
  • Carga contexto completo de la BD en el prompt (trámites, clientes, pagos, gastos, cotizaciones)
  • Potencial problema de escalabilidad con >500 trámites (token overflow)

📜 Reglas de Negocio (Completas)

1. Cálculo Fiscal

  • POST_2017: IGI=10%, DTA=MAX(val×0.8%, $202), PREV=$161, PRV=$0, IVA base=val+IGI+DTA
  • PRE_2016: IGI=50%, DTA=$408 fijo, PREV=$240, PRV=$290, IVA base=val+IGI+DTA
  • AMPARO: precio all-inclusive en MXN, impuestos individuales en $0
  • Validación fuerte: si parámetros fiscales son NULL, lanza error con instrucciones
  • IVA base NO incluye PREV ni PRV (Art. 14-A Ley Aduanera)
  • Parámetros fiscales editables con histórico de vigencia

2. Precios y Catálogo

  • Precio lookup jerárquico: ESPECÍFICO > MARCA > GENÉRICO
  • Rango de antigüedad: 1-12 años (los mayores de 12 se tratan como 12)
  • Match scoring con ajuste por cilindrada del motor
  • Marcas normalizadas vía tabla aliases + pg_trgm similarity
  • Fracción arancelaria auto-clasificada por cilindrada + tipo de vehículo

3. Tipo de Cambio

  • TC referencia = Banxico FIX API (cache diario)
  • TC aplicado = TC referencia + margen (default 0.30)
  • Si Banxico falla, usa último cacheado con flag is_stale
  • Si USD sin TC, error 422

4. Trámites — Reglas Operativas

  • Número consecutivo secuencial por tenant (SELECT MAX + lock)
  • Cobro_total inmutable si ya hay pagos registrados
  • No VERDE_ENTREGADO sin al menos una entrega
  • No COBRADO sin pagos verificados ≥ total
  • AMARILLO asignado automáticamente: VERDE + saldo>0 + 7 días
  • CANCELADO terminal desde cualquier estado (requiere ADMIN)
  • Transiciones hacia atrás requieren ADMIN

5. Pagos — Reglas Financieras

  • Comprobante SIEMPRE obligatorio — cero excepciones
  • Pago USD sin TC registrado nunca se acepta
  • Verificación exclusiva para GERENTE o ADMIN
  • Pago verificado → si suma ≥ cobro_total → trámite a COBRADO
  • Eliminación solo si no verificado (o ADMIN con auditoría)

6. Multi-tenancy

  • TenantId extraído del JWT en cada request
  • Auto-asignado en creación, inmutable en modificación
  • QueryFilter global de EF Core aplica en todas las queries
  • Catálogos globales (aduanas, fracciones, marcas) sin tenant_id

7. Soft Deletes

  • Cliente, Vehículo, Pago, GastoHormiga: borrado lógico con DeletedAt
  • Roles de sistema (EsSistema = true) no se pueden eliminar
  • Cliente con trámites activos no se puede eliminar
  • Vehículo con trámites no se puede eliminar

8. Importador de Datos

  • Idempotente: valida por tenant_id + numero_legacy
  • Transaccional: rollback si algo falla
  • No sobreescribe existentes, reporta duplicados
  • Modo dry-run antes de importación real
  • Normaliza marcas por aliases, clientes por apodo case-insensitive

🔴 Crítico — Secretos expuestos en appsettings.json: Connection string de Neon PostgreSQL, Banxico API Token, Gemini API Key, Cloudflare R2 keys. JWT Secret Key es placeholder inseguro.
🔴 Almacenamiento Cloudflare R2: Configuración lista en JSON pero FileStorageService opera en modo Local. En producción con múltiples instancias los archivos locales se pierden.
🔴 SMTP Mailtrap: Solo configurado para desarrollo (sandbox). Envío de cotizaciones por email no funciona en producción.
🟡 JWT 24h: Expiración de 1440 minutos es excesiva para sistema financiero. El plan original especificaba 15 minutos.
🟡 Sin tests automatizados: Solo existen tests de ejemplo (FiscalCalculatorTests con 7 tests, TramiteStateServiceTests con 7 tests). Karma configurado en frontend pero sin tests funcionales.
🟡 Reportes limitados: Solo conversión de cotizaciones implementado. Faltan: financiero, pipeline, productividad tramitadores, gastos hormiga, estado de cuenta clientes.
🟡 Inconsistencia en máquina de estados: Portal cliente define 16 estados pero TramiteStateService solo valida 6. Los estados extra del portal no tienen transiciones validadas.
🟡 Asignación automática de AMARILLO: Regla de negocio (VERDE + saldo>0 + 7 días) no implementada como background job.
🟡 Búsqueda global Cmd+K: Contemplada en diseño del topbar pero no implementada.
🟡 Notificaciones in-app incompletas: Infraestructura SignalR existe pero las notificaciones como feature (campana en topbar) están incompletas.
🟡 FluentValidation solo en Login: Solo existe LoginRequestValidator. Todos los demás endpoints validan con if/throw inline.
🟡 app.component.html con código muerto: El archivo contiene ~336 líneas del template placeholder de Angular CLI. app.component.ts usa template inline, así que este archivo es código muerto.
🟡 CotizadorService god object: 1,052 líneas que mezclan: NHTSA, lookup precios, cálculo fiscal, CRUD, dashboard, conversión a trámite, generación de folio, reportes.
🟡 Conversión monetaria duplicada: ConvertPagoToMxn existe como método estático en 4 archivos diferentes.
🟡 Enums como strings mágicos: Todos los estados, tipos, métodos de pago como strings raw sin tipado.
🟡 RodriService — system prompt masivo: Carga toda la BD en el prompt (~10 queries con Includes completos). Puede exceder límites del modelo con >500 trámites.
🟡 CORS solo para localhost: Configurado para localhost:4200/4300. No hay URLs de producción.
